# RFC-0012 — Single CHANGELOG owner: release-please (retire the manual [Unreleased] gate)

> **Status:** Under Review
> **Date:** 2026-06-07
> **Author(s):** @valdomirosouza
> **Reviewers:** DevOps Lead (owner of `.github/workflows/`), Release Manager, Tech Lead
> **Related Issue:** #110
> **Related RFC:** RFC-0005/0008 (version-file sync), RFC-0006 (manifest mode)
> **Related ADR:** ADR-0057 (version SoT), ADR-0058
> **Change type:** Normal

---

## 1. Context

`CHANGELOG.md` is managed by **two** mechanisms that conflict:

- The **pr-governance `changelog-updated` job** requires every non-docs PR to add an entry under
  a manual `## [Unreleased]` section (`.github/workflows/pr-governance.yml`).
- **release-please** generates versioned `## [X.Y.Z]` sections from Conventional-Commit titles
  and **prepends** them on each release — it does **not** consume the manual `[Unreleased]`.

Because release-please never clears the manual section, orphaned `## [Unreleased]` blocks
accumulate on `main` (three were present: lines 14, 38, 77), the open release PR (#106) inherits
them, and release notes under-report (the chore/ci/docs work — e.g. the RTK removal and the SBOM
fix — never reaches the generated notes). This was surfaced while verifying open PRs/issues are clean.

## 2. Decision

**release-please is the single owner of `CHANGELOG.md`.** Concretely:

1. **Remove the `changelog-updated` job** from `pr-governance.yml`. The repo already enforces a
   **Conventional-Commit PR title** (`conventional-title` job), which is exactly what release-please
   consumes — so changelog coverage is guaranteed by title discipline, not a manual edit.
2. **Widen `release-please-config.json` `changelog-sections`**: add `refactor` (Refactoring),
   `ci` (CI/CD), `build` (Build & Dependencies), `revert` (Reverts) as **visible** sections so
   non-feat work appears in the notes. `chore` stays hidden (genuine noise); `docs` stays visible.
3. **Reconcile `main`**: strip the orphaned `## [Unreleased]` sections and replace the CHANGELOG
   preamble with a "generated by release-please — do not hand-edit" notice. Pre-2026-06 entries
   keep their prior Keep-a-Changelog format (historical).

After this lands, release-please regenerates the release PR (#106) with a clean, single set of
versioned sections and no stray `[Unreleased]`.

## 3. Alternatives Considered

| Option                                                                | Pros                                                                                           | Cons                                                                          | Why rejected                       |
| --------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | ---------------------------------- |
| A (proposed) — release-please owns it; drop manual gate               | One source of truth; no orphans; notes auto-generated; matches the release tool already in use | `chore`-typed changes don't appear (by design)                                | —                                  |
| B — manual Keep-a-Changelog owns it; disable release-please changelog | Full editorial control of notes                                                                | Fights release-please's core model; loses auto-generation; more manual upkeep | Against the chosen release tooling |
| C — keep both, reconcile periodically                                 | No process change                                                                              | The collision recurs every cycle (the bug we just hit)                        | Doesn't fix root cause             |

## 4. Impact Assessment

| Area                | Impact                    | Notes                                                                                                      |
| ------------------- | ------------------------- | ---------------------------------------------------------------------------------------------------------- |
| API / DB / PII      | None                      | CI/release tooling + docs                                                                                  |
| Release notes       | Positive                  | Single clean source; `refactor`/`ci`/`build` now surfaced                                                  |
| PR authoring        | Simpler                   | No manual `[Unreleased]` edit required; just a conventional title                                          |
| Coverage of changes | Conventional-title-driven | Notable changes should use `feat`/`fix`/`refactor` (not `chore`) to appear                                 |
| Branch protection   | **Action needed**         | If "CHANGELOG updated" is a _required_ check, remove it from branch protection (the job no longer reports) |

**Non-goal:** changing the version SoT (`version.txt`, ADR-0057) or the conventional-title gate.

## 5. Rollout Plan

1. Merge after DevOps/Release-Manager review (touches `.github/workflows/` → human review).
2. Repo admin: remove "CHANGELOG updated" from required status checks if present (see §4).
3. On the next push, release-please regenerates the release PR with clean notes + widened sections.
4. Smoke test: open a `ci:`/`refactor:` PR with no CHANGELOG edit → it is **not** blocked, and the
   change appears in the next release PR's notes.

## 6. Rollback Plan

Revert this PR — restores the `changelog-updated` job and the prior `changelog-sections`. The
stripped `[Unreleased]` sections are recoverable from git history. No data loss (all content is in
commit history).
