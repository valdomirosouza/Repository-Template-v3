# ADR-0020 — FinOps: LLM Cost Allocation and Budget Enforcement

**Status:** Accepted
**Date:** 2026-05-28
**Authors:** Tech Lead, SRE Lead
**Spec:** N/A — operational policy
**Supersedes:** None | **Superseded by:** None

---

## Context

The system makes LLM API calls (Anthropic) on behalf of users. Each call consumes
tokens that are billed per-million. Without explicit budget controls:

1. A runaway self-reflection loop in the harness could exhaust the monthly budget
   in hours, causing service degradation for all users.
2. There is no attribution of cost to `action_type`, making it impossible to
   identify which agent behaviours are economically unsustainable.
3. FinOps review has no actionable data — only a monthly invoice total.

`src/observability/metrics.py` already defines `llm_tokens_total`, `llm_tokens_budget_total`,
and `agent_cost_per_resolution_tokens`, but no policy governs how they are enforced.

---

## Decision

### 1. Budget configuration

A single monthly token budget is configured per service instance:

```
LLM_MONTHLY_TOKEN_BUDGET=1_000_000   # default; override per environment
COST_ALERT_THRESHOLD_USD=100.0       # advisory alert (not a hard stop)
```

`init_budget_gauge(service, monthly_token_budget)` is called at startup and exposes
the budget as `llm_tokens_budget_total{service}` for Prometheus ratio queries.

### 2. Budget enforcement tiers

| Consumption | Action                                                                                                   | Metric                                       |
| ----------- | -------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| < 80%       | Normal operation                                                                                         | —                                            |
| ≥ 80%       | `LLMTokenBudgetExceeded90Percent` alert fires (see `agent-alerts.yaml`)                                  | `llm_tokens_budget_total` ratio              |
| ≥ 95%       | Circuit breaker threshold lowered from 5 to 1 consecutive failure — effectively blocks new LLM calls     | `llm_circuit_breaker_threshold` env override |
| 100%        | On-call manually disables the `autonomous-mode` feature flag; HITL-only mode continues without LLM calls | Feature flag, ADR-0015                       |

The 100% enforcement is intentionally manual: an automated hard stop could drop
in-flight HITL requests and violate audit log completeness. Human decision is
required to determine whether to extend budget or accept service degradation.

### 3. Cost attribution

Every `record_llm_call()` call in `metrics.py` tags tokens with `service` and `model`.
Every `record_agent_performance()` call tags `action_type`.

**Allocation report** (weekly, generated by `make agent-feedback-check` extended query):

```promql
# Cost per action_type over 7 days
sum(increase(agent_cost_per_resolution_tokens_sum[7d])) by (action_type)
/ sum(increase(agent_cost_per_resolution_tokens_count[7d])) by (action_type)
```

Action types with p99 token cost > 10 000 tokens trigger `AgentCostPerResolutionHigh`
alert and must be reviewed for prompt efficiency before the next sprint.

### 4. Harness cost controls

The harness enforces token discipline:

- `harness_max_iterations` caps the Generator–Evaluator retry loop (default 15)
- `llm_token_budget_per_request` caps tokens per individual LLM call (default 2 000)
- `harness_patch_proposal_threshold` limits self-reflection cycles (default 2)

Exceeding `harness_max_iterations` without passing escalates to HITL — the human
operator decides whether to continue (and spend more tokens) or abort.

### 5. SessionMemory Redis encryption (deferred from ADR-0019)

Sprint context in `SessionMemory` (`src/memory/session_memory.py`) is stored in Redis
without value-level encryption. Post-PII-masking sprint context contains no L1/L2 PII
by invariant, so the risk is lower than HITL payloads (ADR-0019). Encryption will be
added in a future sprint when `EncryptedField` performance on high-frequency short-lived
keys is benchmarked. Until then, Redis TLS (ADR-0019) provides transport-layer protection.

---

## Consequences

### Positive

- Token consumption is observable per `action_type` — unsustainable agent behaviours
  are identifiable before they exhaust the budget
- Three-tier enforcement gives operators time to react before hard service degradation
- Harness iteration caps prevent single runaway tasks from consuming disproportionate budget

### Negative / Trade-offs

- The 95% threshold (circuit breaker tightening) is a heuristic — a spike in legitimate
  traffic near month-end could trigger it prematurely
- Monthly budget is a single global cap; there is no per-`action_type` sub-budget
  (accepted trade-off: added complexity not justified at current scale)

---

## Alternatives Considered

**Hard stop at 100% via middleware**
Rejected: drops in-flight requests; violates audit log completeness requirement (ADR-0012).

**Per-request cost estimation pre-call**
Rejected: token counts are not precisely predictable before the call; estimation errors
would either under-block or excessively throttle.

**External FinOps tool (e.g. Langfuse, Helicone)**
Deferred: adds a third-party dependency and egress of LLM metadata. Revisit when
multi-model support is needed.

## Appendix — Agentic ROI Model (added 2026-06-05, Issue #7)

### Cost per Task Formula

```
cost_per_task = (tokens_input + tokens_output) × price_per_million_tokens / 1_000_000
roi_ratio     = engineer_hourly_rate × avg_task_hours / cost_per_task
```

At current pricing (claude-sonnet-4-6: $3/$15 per M tokens) and a median task consuming
~8,000 tokens, `cost_per_task ≈ $0.14`. An engineer-hour at $120 makes the ROI ratio ~857×
for a 10-minute task that would otherwise take an hour.

### Net-New Work Multiplier

Per the _2026 Agentic Coding Trends Report_, ~27% of AI-assisted work is additive —
tasks that would not have been undertaken without agent capacity. Budget allocation
should account for this multiplier: the effective throughput gain exceeds the raw
velocity improvement.

### Metrics (added by Issue #7)

The following Prometheus metrics now instrument this model:

| Metric                                                     | Purpose                                     |
| ---------------------------------------------------------- | ------------------------------------------- |
| `agent_session_tasks_total{task_type, outcome}`            | Counts planned vs net_new vs papercut tasks |
| `agent_session_duration_seconds`                           | Measures wall-clock time per session        |
| `agent_cycle_time_seconds{stage}`                          | Tracks spec-to-green-CI lead time           |
| `agent_subtask_duration_seconds{agent_role, harness_mode}` | Per-specialization latency                  |
| `agent_subtask_error_total{agent_role, error_type}`        | Per-specialization error rate               |

These feed the `agent-productivity` Grafana dashboard
(`infrastructure/monitoring/grafana/dashboards/agent-productivity.json`).

### Budget Allocation for Formerly Non-Viable Work

Projects that were previously below the viability threshold (cost > expected value)
may now be viable with agent augmentation. A quarterly FinOps review should include
a "net-new work" category — budget earmarked for tasks that agents unlock rather than
accelerate. Recommended starting allocation: 15% of the total LLM budget.
