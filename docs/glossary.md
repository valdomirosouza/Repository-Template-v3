## 1. AI Governance & Agents

| Term                                    | Definition                                                                                                                                                                                                                                                                                                                                                                                    |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Agent**                               | An autonomous software component that perceives inputs, reasons using an LLM, and acts to achieve a goal. Follows the Perception → Reason → Act loop pattern. All agent actions with real-world effects route through `src/agents/hitl_gateway.py`.                                                                                                                                           |
| **Autonomy Boundary**                   | The defined line between HOTL (autonomous) and HITL (human-gated) execution. Documented in `docs/ai-governance/autonomy-boundaries.md`. Crossing this boundary for a new action type requires explicit governance approval.                                                                                                                                                                   |
| **Autonomy Level**                      | One of six ordered states controlling how much autonomy an agent has: `NONE` → `READ_ONLY` → `TESTS_ONLY` → `LOW_RISK` → `MEDIUM_RISK` → `FULL`. Controlled by the `autonomous-mode` feature flag. Escalating beyond `LOW_RISK` requires ADR-0015 approval.                                                                                                                                   |
| **BiasProvider**                        | A protocol (`src/agents/risk_scorer.py`) that supplies per-action-type bias offsets computed by `FeedbackLoop`. The `RiskScorer` calls `get_bias(action_type)` to adjust raw scores before routing.                                                                                                                                                                                           |
| **BugHistoryStore**                     | A memory layer (`src/agents/memory/`) that records HITL rejection events as vector documents so future agents can retrieve similar past rejections before repeating risky actions. Backed by pgvector; TTL 90 days.                                                                                                                                                                           |
| **DocumentIndexer**                     | Scans `specs/` and `docs/adr/` and upserts each file as a `VectorDocument`, enabling semantic recall of spec and ADR content during agent reasoning. Triggered by CI on push to `main` or manually via `make memory-index`.                                                                                                                                                                   |
| **Embedder**                            | A protocol for converting text to a float vector representation (`list[float]`). `StubEmbedder` is used in tests; a production embedder is caller-provided. Embedding is always computed from PII-masked text.                                                                                                                                                                                |
| **Evaluator**                           | The third agent in the multi-agent harness loop. Scores `Generator` output against the `SprintContract` criteria using LLM-as-judge; returns PASS, RETRY, PATCH, or ESCALATE.                                                                                                                                                                                                                 |
| **FeedbackLoop**                        | A background component (`src/agents/feedback_loop.py`) that periodically queries Prometheus rejection metrics, computes per-`action_type` bias adjustments, and publishes them to Kafka topic `agent.feedback.applied`. Runs every 300 s (configurable).                                                                                                                                      |
| **Generator**                           | The second agent in the harness loop. Receives the `TaskBrief` from the Planner and produces a concrete artefact (code, plan, document). Operates under the harness_mode resource envelope.                                                                                                                                                                                                   |
| **Harness**                             | The multi-agent orchestration framework (`src/agents/harness/`) that runs Planner → Generator → Evaluator in a loop until the Evaluator issues PASS or max retries are exceeded. See `specs/ai/harness-design.md`.                                                                                                                                                                            |
| **harness_mode**                        | One of three resource/cost envelopes for harness execution: `solo` (1×, Generator only), `simplified` (5–10×, Generator + Evaluator), `full` (15–25×, Planner + Generator + Evaluator). Selected per sprint based on complexity.                                                                                                                                                              |
| **HITL**                                | Human in the Loop — an oversight model where an agent proposes an action and a human must explicitly approve it before execution. Implemented in `src/agents/hitl_gateway.py`. **No auto-approval on timeout** — timed-out requests are rejected, not auto-approved.                                                                                                                          |
| **HITL Gateway**                        | The central enforcement point (`src/agents/hitl_gateway.py`) for all consequential agent actions. Routes actions to HITL or HOTL based on `risk_score` vs. `hitl_risk_threshold`. Dual ownership: Security + AI Governance.                                                                                                                                                                   |
| **HOTL**                                | Human on the Loop — an oversight model where an agent acts autonomously while a human monitors with override capability. Used for read-only and low-risk operations. HOTL actions still emit audit log entries and trigger the override window notification.                                                                                                                                  |
| **HOTL Override Window**                | A configurable time window (default 5 min) after a HOTL action executes during which a human reviewer may cancel or roll back the action. Notifications delivered via `HITLNotificationService`.                                                                                                                                                                                              |
| **LLM**                                 | Large Language Model — a neural network trained on large text corpora, used in this system for generation, classification, and reasoning within the agent Reason step. Primary integration: Anthropic Claude via `src/agents/llm_client.py`.                                                                                                                                                  |
| **Orchestrator**                        | The component (`src/agents/orchestrator/orchestrator.py`) that drives a single agent's Perception → Reason → Act loop. Composes `RiskScorer`, `HITLGateway`, `AuditLogger`, and optionally all three memory layers.                                                                                                                                                                           |
| **Planner**                             | The first agent in the `full` harness mode. Decomposes a `TaskBrief` into a structured execution plan that the `Generator` follows. Absent in `solo` and `simplified` modes.                                                                                                                                                                                                                  |
| **PromptInjection Guard**               | Layer 2 of the guardrail pipeline (`src/guardrails/prompt_injection_guard.py`). Detects and blocks adversarial input patterns that attempt to override system instructions before text reaches the LLM. Must never be removed or weakened.                                                                                                                                                    |
| **RAG**                                 | Retrieval-Augmented Generation — an LLM pattern that retrieves relevant documents from a vector store and includes them in the LLM context to ground responses in factual, up-to-date data. Used in agent memory recall.                                                                                                                                                                      |
| **RiskComponents**                      | A frozen dataclass holding the five factor scores used to compute a single `risk_score`: `irreversibility` (w=0.35), `external_effect` (w=0.25), `scale` (w=0.20), `data_sensitivity` (w=0.15), `rejection_rate` (w=0.05).                                                                                                                                                                    |
| **RiskScorer**                          | A deterministic, weight-based scorer (`src/agents/risk_scorer.py`) that computes a `risk_score ∈ [0, 1]` from five factors plus an optional `BiasProvider` offset. Replaces ad-hoc thresholds with an auditable formula.                                                                                                                                                                      |
| **SandboxExecutor**                     | The only sanctioned execution environment (`src/agents/sandbox_executor.py`) for agent-generated code. Running agent code outside the sandbox requires explicit HITL approval (ADR-0016).                                                                                                                                                                                                     |
| **SessionMemory**                       | A short-term Redis-backed key-value store scoped to an agent session (`src/agents/memory/`). Default TTL: 24 h. Enables context continuity across context-reset events within a sprint.                                                                                                                                                                                                       |
| **SprintContract**                      | The acceptance criteria document attached to a `TaskBrief` that the Evaluator uses to score Generator output. A sprint is considered complete only when the Evaluator returns PASS against the SprintContract.                                                                                                                                                                                |
| **TaskBrief**                           | The input document handed to the harness at the start of a sprint. Contains: goal, constraints, acceptance criteria, harness_mode, and max_retries. Maps to a GitHub Issue + linked spec.                                                                                                                                                                                                     |
| **VectorDocument**                      | The schema for a stored memory entry: `{id, content, embedding, source, tags, created_at}`. Content is always PII-masked before storage. Source is one of: `spec`, `adr`, `hitl_rejection`, `sprint_outcome`.                                                                                                                                                                                 |
| **VectorStore**                         | A semantic similarity search layer over indexed documents. `InMemoryVectorStore` (tests/dev), `PostgresVectorStore` (production via pgvector). Protocol-based — implementation is injected, not imported directly.                                                                                                                                                                            |
| **agent_action_v1**                     | The strict structured-output envelope the Reason stage requires from the LLM (`src/agents/schemas/agent_action_v1.py`): `schema_version, intent, action_type, tool_name, target_system, target_environment, operation, parameters, data_classification, external_effect, reversible, compensating_action, agent_confidence`. Validated fail-closed; invalid output routes to HITL (ADR-0054). |
| **ActionSchemaValidator**               | `src/agents/action_schema_validator.py`. Validates an action payload against its per-`action_type` schema before HITL/execution, rejecting hallucinated or malformed payloads (defense-in-depth, ADR-0050).                                                                                                                                                                                   |
| **BehavioralMonitor**                   | `src/agents/behavioral_monitor.py`. Tracks an agent's historical action distribution and flags proposals outside its behavioural envelope (possible drift or prompt injection). Secure-by-Design Pillar 3 (ADR-0049).                                                                                                                                                                         |
| **CompensationRegistry**                | `src/agents/compensation_registry.py`. Resolves an action's compensating (undo) action and answers `can_run_under_hotl()`; non-reversible or over-ceiling actions cannot execute autonomously under HOTL (ADR-0055).                                                                                                                                                                          |
| **Context Graph**                       | `src/agents/context_graph.py`. A runtime session goal-graph (goals, constraints, gathered context, decisions) persisted as JSONB for goal-state continuity across sessions (Gartner Autonomy L4, ADR-0041). Distinct from `.agent/context-graph.json`, the repo-structure bootstrap map (ADR-0057).                                                                                           |
| **Context Seal**                        | `src/agents/harness/context_seal.py`. A tamper-evident seal over a handed-off spec/context; a failed verification (`ContextTamperingError`) blocks sprint execution and escalates to HITL (Secure-by-Design SD2, ADR-0047).                                                                                                                                                                   |
| **Gartner Autonomy Levels**             | The Gartner maturity scale for agentic autonomy (Levels 1–5). This repo targets graduated autonomy, with Level 4 (durable goal-state via the Context Graph) gated behind governance approval.                                                                                                                                                                                                 |
| **HOTL Monitor**                        | `src/agents/hotl_monitor.py`. Drives the HOTL post-execution lifecycle: emits the reviewer notification within the SLO (default 60 s) and opens the override window (ADR-0055).                                                                                                                                                                                                               |
| **Mandatory HITL**                      | A deterministic policy layer (`src/agents/action_policy.py`) evaluated BEFORE the numeric risk score. Categories such as production writes, financial transactions, credential rotation, deploys, and bulk operations always require HITL — a low score cannot downgrade them (ADR-0053).                                                                                                     |
| **Model Behavioral Contract**           | Real-LLM tests (`tests/model_contract/`) asserting the model honours its spec contract (allowed/prohibited actions, refusals, PII non-leakage). A model version cannot be promoted in the dependency manifest without passing them (ADR-0051).                                                                                                                                                |
| **Override Service**                    | `src/agents/override_service.py`. Enforces the HOTL override window and runs the compensation flow (override.requested → compensation.started → succeeded/failed → escalation/confirmed); a failed/absent compensation raises an escalation event (ADR-0055).                                                                                                                                 |
| **Reversibility / Compensating Action** | Per-tool metadata in `tools.yaml`: `reversible` marks whether an action can be undone; `compensating_action` names the tool that reverses it. Drives the HOTL reversibility gate (ADR-0055).                                                                                                                                                                                                  |
| **Risk Calibration**                    | A golden dataset (`tests/unit/agents/test_risk_calibration.py`) pinning scenario → expected risk score → routing tier. CI fails if a scoring change shifts routing without an explicit recalibration (ADR-0054).                                                                                                                                                                              |
| **RuntimePolicyGateway**                | `src/agents/runtime_policy_gateway.py`. A declarative, hot-reloadable policy layer returning ALLOW / REQUIRE_HITL / BLOCK, evaluated at the act layer before the HITL gateway (ADR-0049).                                                                                                                                                                                                     |
| **SpecContractEnforcer**                | `src/agents/spec_contract_enforcer.py`. Injects the agent's permission boundary (`[SPEC_CONTRACT]`) into the system prompt and validates proposed actions against `allowed_action_types`, raising `SpecViolationError` on breach (Secure-by-Design SD1, ADR-0047).                                                                                                                            |
| **Sub-Agent Registry**                  | `src/agents/harness/sub_agent_registry.py`. Catalog of specialised sub-agent configurations the harness may delegate to (ADR-0032).                                                                                                                                                                                                                                                           |
| **Tool Registry**                       | The governed catalog (`src/agents/tool_registry.py`; canonical source `infrastructure/agent-tools/tools.yaml`) of every tool an agent may invoke — with risk level, HITL requirement, execution mode, rate limits, owner team, and reversibility metadata. Unregistered tools are blocked at runtime (ADR-0039 / 0048 / 0053).                                                                |
| **ToolExecutor**                        | `src/agents/tool_executor.py`. The single execution choke-point running the 10-step enforcement sequence (registry assertion → schema → requires-HITL → autonomy permission → rate limit → sandbox routing → execute → pre/post/failure audit) before any action runs (ADR-0053).                                                                                                             |

## 2. Privacy & Data Protection

| Term                   | Definition                                                                                                                                                                                                                                                               |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **CPF**                | Cadastro de Pessoas Físicas — the Brazilian individual taxpayer registry number. L1 Critical PII. Must always be tokenised or masked; never logged or transmitted in plaintext.                                                                                          |
| **Data Masking**       | The process of replacing sensitive values with safe tokens before they are logged, transmitted, or used as LLM input. Implemented in `src/guardrails/pii_filter.py`. Three interception points: API ingress, message broker publish, LLM prompt construction.            |
| **Data Retention**     | Policy controlling how long data is kept before deletion. For this system: vector memory docs 90 days, audit events archived after 1 year, hard-deleted after 5 years. Enforced by `src/jobs/retention_job.py` (daily cron).                                             |
| **Data Subject**       | A natural person whose personal data is processed by the system. Under LGPD and GDPR, data subjects have rights including access, correction, and erasure. Erasure requests must be fulfilled within 15 days.                                                            |
| **Erasure Request**    | A data subject's request to delete all personal data held by the system. Must be fulfilled within 15 days of receipt. Affects all records where the data subject's identifier appears (vector store, audit events, session cache).                                       |
| **L1 – Critical PII**  | The highest PII sensitivity tier. Includes CPF/SSN, health records, biometric data, financial credentials, precise geolocation. Requires encryption at rest and in transit; access strictly audited. Tagged as `[CPF]`, `[SSN]`, `[HEALTH]`, `[CARD]`.                   |
| **L2 – Sensitive PII** | Name, email address, phone number, IP address, date of birth. Requires masking before logging and must not appear in LLM prompts. Tagged as `[EMAIL]`, `[PHONE]`.                                                                                                        |
| **L3 – Internal PII**  | Username, session token, organisation role. May be used internally by services but must not be logged in plaintext or exposed externally. Tagged as `[TOKEN]`.                                                                                                           |
| **L4 – Public**        | Declared role, organisation name, publicly stated identifiers. No special handling required beyond standard security practices.                                                                                                                                          |
| **PII**                | Personally Identifiable Information — any data that can directly or indirectly identify a natural person. Classified L1–L4 in `docs/privacy/pii-inventory.md`; masked at three interception points per ADR-0012.                                                         |
| **PII Filter**         | Guardrail layer 1 (`src/guardrails/pii_filter.py`). Every write to any memory layer, every message broker publish, and every LLM call must pass `pii_filter.mask_text()` or `pii_filter.mask_dict()`. The entry point to the guardrail pipeline.                         |
| **Privacy by Design**  | The principle that privacy safeguards are baked into system architecture from the start rather than added as an afterthought. Embodied in the three mandatory PII interception points and the DPIA/RIPD gate.                                                            |
| **RoPA**               | Record of Processing Activities — the GDPR Art. 30 register of all data processing activities maintained by the data controller. Template at `docs/privacy/data-processing-register.md`. Must be updated whenever a new processing activity is introduced.               |
| **Tokenisation**       | Replacing a sensitive value with a non-sensitive surrogate (token) that can be mapped back to the original only via a separate, secured token vault. Used for L1 PII such as CPF numbers. Distinct from encryption (token has no mathematical relationship to original). |

## 3. Compliance & Legal

| Term                 | Definition                                                                                                                                                                                                                                                                                                                                                   |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **ADR**              | Architecture Decision Record — a document capturing a significant architectural decision, its context, options considered, and consequences. ADRs in `docs/adr/` are binding on all implementation unless superseded by a newer ADR.                                                                                                                         |
| **CAB**              | Change Advisory Board — the governance body that reviews and approves Normal and Emergency changes. Composition: Tech Lead, Security Lead, SRE Lead, and DPO (for privacy-impacting changes).                                                                                                                                                                |
| **DPIA**             | Data Protection Impact Assessment — a structured risk assessment required by GDPR Art. 35 before any high-risk personal data processing begins. Template at `docs/privacy/dpia/dpia-v1.md`. Must be signed by the DPO before merge.                                                                                                                          |
| **DPO**              | Data Protection Officer — the person responsible for overseeing LGPD and GDPR compliance. Required approver for any change that introduces or modifies personal data processing. Co-author on all DPIA and RIPD documents.                                                                                                                                   |
| **GDPR**             | General Data Protection Regulation (EU 2016/679) — the EU data privacy law governing processing of personal data of EU residents. Extraterritorial — applies whenever EU residents' data is processed regardless of where the processor is located.                                                                                                          |
| **LGPD**             | Lei Geral de Proteção de Dados (Lei 13.709/2018) — Brazil's federal data privacy law governing processing of personal data of Brazilian residents. Similar scope to GDPR; applies to any organisation processing data of people in Brazil.                                                                                                                   |
| **NIST**             | National Institute of Standards and Technology — the US standards body whose frameworks (NIST CSF, NIST SP 800-53, NIST AI RMF) inform the security and AI governance controls in this repository.                                                                                                                                                           |
| **OWASP**            | Open Web Application Security Project — the nonprofit producing the OWASP Top 10 web security risk list and the OWASP LLM Top 10 (the equivalent for LLM-powered applications). Referenced in `skills/ai/guardrails.md`.                                                                                                                                     |
| **OWASP LLM Top 10** | The OWASP list of the ten most critical security risks in LLM applications: LLM01 Prompt Injection, LLM02 Insecure Output Handling, LLM03 Training Data Poisoning, LLM04 Model DoS, LLM05 Supply Chain Vulnerabilities, LLM06 Sensitive Information Disclosure, LLM07 Insecure Plugin Design, LLM08 Excessive Agency, LLM09 Overreliance, LLM10 Model Theft. |
| **RIPD**             | Relatório de Impacto à Proteção de Dados Pessoais — the LGPD Art. 38 equivalent of a DPIA. Required before any production release that introduces or changes personal data processing for Brazilian residents.                                                                                                                                               |
| **RFC**              | Request for Change — a formal document proposing a Normal or Emergency change for review by the CAB. Template at `docs/change-management/RFC-TEMPLATE.md`. Normal changes require 5 business days; Emergency changes use an expedited path.                                                                                                                  |
| **SDD**              | Spec-Driven Development — the practice in this repository where a spec in `specs/` is written and approved before any implementation begins. No PR may be merged without referencing a spec ID. See `skills/sdlc/spec-lifecycle.md`.                                                                                                                         |
| **SLSA**             | Supply-chain Levels for Software Artifacts (slsa.dev) — a security framework for software supply chain integrity. Target for this project: SLSA Level 2+ (provenance generated and signed in CI using Cosign).                                                                                                                                               |
| **EU AI Act**        | The EU Artificial Intelligence Act (Regulation 2024/1689) — risk-tiered AI regulation. Drives the agent-disclosure transparency (Art. 13), human-oversight (Art. 14), and post-market monitoring (Art. 72) requirements in this repo.                                                                                                                        |
| **ISO 27001**        | The international standard for Information Security Management Systems (ISMS). Governs the three-tier change-management process (standard / normal / emergency) and append-only change-evidence records (ADR-0027).                                                                                                                                          |
| **SOX**              | Sarbanes–Oxley Act — US law mandating financial-reporting integrity controls. When applicable: immutable audit records on financial-data write paths, ≥7-year retention, and segregation of duties (ADR-0026).                                                                                                                                               |

## 4. Security

| Term                   | Definition                                                                                                                                                                                                                                                                                       |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Audit Logger**       | Guardrail layer 4 (`src/guardrails/audit_logger.py`). Records every agent action as an immutable, append-only entry. Write failures block the associated action. Ownership: Security Lead; changes require Security review.                                                                      |
| **Bandit**             | A Python static analysis tool that identifies common security issues (hardcoded secrets, use of `eval`, `pickle`, `subprocess`, etc.). Run as part of the pre-commit hook and CI SAST gate. Configuration in `pyproject.toml [tool.bandit]`.                                                     |
| **Cosign**             | A container image signing tool (from the Sigstore project) used in CI to sign Docker images and SLSA provenance attestations. Enables supply chain verification at deploy time.                                                                                                                  |
| **detect-secrets**     | A Yelp open-source tool that scans all files for high-entropy strings and known secret patterns before they are committed. Run as a pre-commit hook; blocks commits containing potential secrets.                                                                                                |
| **HMAC-SHA256**        | Hash-based Message Authentication Code using SHA-256 — the algorithm used to sign HITL notification webhook payloads (`X-HITL-Signature: sha256=<hmac>`). Recipients must verify the signature before processing any decision callback.                                                          |
| **JWT**                | JSON Web Token — a compact, URL-safe token format used for API authentication. Stateless; signature verified on each request. Must be transmitted only over TLS.                                                                                                                                 |
| **mTLS**               | Mutual TLS — a variant of TLS where both client and server authenticate with certificates. Used for internal service-to-service communication to prevent spoofing.                                                                                                                               |
| **pip-audit**          | A tool that audits Python environments against the PyPI Advisory Database (OSV) for known vulnerabilities in installed packages. Run in CI on every dependency update.                                                                                                                           |
| **SAST**               | Static Application Security Testing — automated source code analysis for security vulnerabilities without executing the program. Tools: Bandit (Python security), Ruff (code quality + security rules), detect-secrets (credential scanning). SAST gates must never be bypassed (`--no-verify`). |
| **SBOM**               | Software Bill of Materials — a formal inventory of all software components, their versions, and their licences. Generated by Syft in CI and signed with Cosign. Required for supply chain transparency and vulnerability tracking.                                                               |
| **Syft**               | An open-source SBOM generation tool (Anchore). Produces CycloneDX or SPDX format SBOMs from container images or filesystems. Run in CI after every Docker image build.                                                                                                                           |
| **TLS**                | Transport Layer Security — the cryptographic protocol that provides encrypted, authenticated communication over the network. All external-facing endpoints must use TLS 1.2 or higher.                                                                                                           |
| **Abuse Case Testing** | Adversarial tests (`tests/abuse_cases/`) attempting jailbreaks, goal hijacking, context overflow, multi-agent trust abuse, and spec-boundary violations. The count must never decrease (ADR-0050).                                                                                               |
| **Checkov**            | An IaC static-analysis tool that scans Terraform / K8s / Helm for misconfigurations. Blocking gate on `infrastructure/` changes.                                                                                                                                                                 |
| **Checkstyle**         | A Java source-style / lint tool run in the `CI — Java` gate for services under `services/`.                                                                                                                                                                                                      |
| **CodeQL**             | GitHub's semantic code-analysis engine for security vulnerabilities; runs as the `CodeQL Analysis` workflow.                                                                                                                                                                                     |
| **CycloneDX**          | An OWASP SBOM format emitted by Syft and attested with Cosign.                                                                                                                                                                                                                                   |
| **DAST**               | Dynamic Application Security Testing — runtime black-box scanning (OWASP ZAP) run in staging as a blocking gate before production promotion.                                                                                                                                                     |
| **DevSecOps**          | Integrating security controls (SAST, SCA, IaC scan, DAST, SBOM, signing) directly into the CI/CD pipeline. See `skills/devsecops/`.                                                                                                                                                              |
| **gosec**              | A Go security static-analysis tool, run in the Go CI lint gate.                                                                                                                                                                                                                                  |
| **MLSecOps**           | Security operations for ML/LLM systems — model behavioural contracts, abuse-case testing, and behavioural monitoring layered onto DevSecOps (Secure-by-Design, ADR-0047–0051).                                                                                                                   |
| **NVD**                | National Vulnerability Database (NIST) — the CVE feed OWASP dependency-check queries. An `NVD_API_KEY` is recommended to avoid heavy throttling.                                                                                                                                                 |
| **SpotBugs**           | A Java static-analysis tool (successor to FindBugs) run in the `CI — Java` lint gate.                                                                                                                                                                                                            |
| **Trivy**              | An open-source vulnerability and misconfiguration scanner (Aqua) for container images and filesystems; blocks on CRITICAL CVEs.                                                                                                                                                                  |
| **Zero-Trust Tooling** | The principle that an agent may invoke only registered tools, each gated by autonomy level, sandbox mode, and rate limits — default-deny at the execution choke-point (ADR-0048).                                                                                                                |

## 5. SRE & Reliability

| Term                                     | Definition                                                                                                                                                                                                                                                                |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Blue-Green Deploy**                    | A deployment strategy maintaining two identical production environments (blue = current, green = new). Traffic is shifted to green atomically after validation. Enables instant rollback by redirecting traffic back to blue.                                             |
| **Canary Deploy**                        | A deployment strategy where a new version receives a small percentage of traffic (5% → 25% → 100%) while Golden Signals are monitored at each step. Automated rollback triggers on SLO breach.                                                                            |
| **CUJ**                                  | Critical User Journey — a key end-to-end user workflow with defined SLO targets for availability and latency. Each CUJ has a dedicated spec file in `docs/sre/cuj/` and a Grafana dashboard. Regression in a CUJ SLO triggers an error budget alert.                      |
| **DLQ**                                  | Dead Letter Queue — a message broker queue that receives events that could not be processed by the primary consumer after all retry attempts are exhausted. Monitored as a Golden Signal (Saturation). Alerts fire when DLQ depth exceeds threshold.                      |
| **Error Budget**                         | The permitted amount of unreliability within an SLO window, expressed as downtime minutes or failed request allowance. Formula: `(1 − SLO_target) × window_duration`. Feature freeze is triggered when budget remaining falls below 10%.                                  |
| **Golden Signals**                       | The four key SRE observability metrics (Google SRE Book): **Traffic** (request rate), **Error Rate** (% non-2xx), **Saturation** (queue depth, CPU, memory), **Latency** (p50/p95/p99). Dashboards in `infrastructure/monitoring/grafana/dashboards/golden-signals.json`. |
| **HPA**                                  | Horizontal Pod Autoscaler — a Kubernetes controller that automatically scales the number of pod replicas based on observed CPU, memory, or custom metrics. Configured in `infrastructure/k8s/`.                                                                           |
| **MTTR**                                 | Mean Time To Recovery — the average time from when an incident is detected to when service is fully restored. A key operational health metric; tracked alongside MTTD to measure response effectiveness.                                                                  |
| **MTTD**                                 | Mean Time To Detect — the average time between an incident starting and it being detected. Reduced by comprehensive alerting on Golden Signals and structured log anomaly detection.                                                                                      |
| **PRR**                                  | Production Readiness Review — a mandatory checklist completed and signed off by SRE Lead, Tech Lead, Security Lead, and DPO before every production deployment. Template at `docs/sre/prr/PRR-TEMPLATE.md`. No deploy proceeds without a signed PRR.                      |
| **SLI**                                  | Service Level Indicator — a quantitative measurement of a specific service behaviour (e.g., `success_rate = non_5xx_requests / total_requests`). SLIs are the raw measurements that SLOs are set against.                                                                 |
| **SLO**                                  | Service Level Objective — a target value or range for an SLI over a defined window (e.g., 99.9% availability over 30 days). Defined in `docs/sre/slo/slo.yaml`. SLO breaches trigger error budget alerts and may freeze feature work.                                     |
| **SRE**                                  | Site Reliability Engineering — the discipline of applying software engineering principles to operations, reliability, and scalability. Key practices in this project: Golden Signals, SLO-driven alerting, error budgets, PRR, CUJs, and toil elimination.                |
| **TOIL**                                 | Operational work that is manual, repetitive, automatable, and scales with service growth without producing lasting value. SRE principle: keep toil below 50% of each engineer's time. Tracked and systematically automated over time.                                     |
| **DORA**                                 | DevOps Research and Assessment — the four delivery-performance metrics: Deployment Frequency, Lead Time for Changes, Change Failure Rate, and Time to Restore (MTTR). Targets and tracking in CLAUDE.md §12 / ADR-0028.                                                   |
| **Liveness / Readiness / Startup Probe** | Kubernetes health checks: `startupProbe` gates slow-start containers, `readinessProbe` gates traffic, `livenessProbe` restarts hung pods. Required on every Deployment (ADR-0042, `skills/sre/probe-strategy.md`).                                                        |
| **Pushgateway**                          | A Prometheus component that accepts pushed metrics from short-lived / batch jobs (e.g. the CI DORA deployment event) for later scraping.                                                                                                                                  |

## 6. Observability

| Term                     | Definition                                                                                                                                                                                                                                                |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Jaeger**               | An open-source distributed tracing backend (CNCF project). Receives traces from OpenTelemetry collectors and provides a UI for querying spans and trace waterfalls. Default endpoint: `http://localhost:16686`.                                           |
| **Log**                  | Structured JSON output emitted via `structlog`. Every log line includes `trace_id` and `span_id` for correlation with distributed traces. PII-masked before emission. Collected by the OTel collector and forwarded to the logging backend.               |
| **Metric**               | A time-series measurement exposed via the Prometheus client library (`prometheus_client`). Standard naming: `<service>_<noun>_<unit>_total` for counters. All agent guardrail layers expose standard metrics.                                             |
| **OTel / OpenTelemetry** | An open-source observability framework providing standardised APIs and SDKs for traces, metrics, and logs. Bootstrapped in `src/observability/otel_setup.py`. Vendor-neutral: exporters send to Jaeger (traces), Prometheus (metrics), and a log backend. |
| **Prometheus**           | An open-source monitoring system and time-series database. Scrapes metric endpoints via pull model. Used to store all Golden Signals; also the data source for FeedbackLoop rejection rate queries.                                                       |
| **Span**                 | A single named, timed operation within a distributed trace. Spans form a parent-child tree within a trace. Every external call, DB query, and agent action should be wrapped in a span.                                                                   |
| **structlog**            | A Python structured logging library. Used throughout the codebase to emit machine-parseable JSON logs. Context variables (trace_id, agent_id, session_id) are bound once and propagated automatically.                                                    |
| **Trace**                | A correlated record of a request's execution path across all services, collected via OpenTelemetry and stored in Jaeger. Every trace carries a `trace_id` propagated via the W3C TraceContext standard (`traceparent` header).                            |
| **W3C TraceContext**     | The W3C standard for propagating distributed trace context across HTTP and message broker boundaries via `traceparent` and `tracestate` headers. Ensures trace continuity from API ingress through async Kafka consumers.                                 |
| **Grafana**              | The visualisation / dashboarding layer over Prometheus and the trace / log backends. Hosts the Golden Signals, DORA, and agent-feedback dashboards in `infrastructure/monitoring/grafana/`.                                                               |

## 9. Development Practices & SDLC

| Term                       | Definition                                                                                                                                                                                                                                                                |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **CHANGELOG**              | A structured log of notable changes per version following the Keep a Changelog format. Every production change must be recorded under the correct category (Added, Changed, Fixed, Security, Deprecated, Removed). Located at `CHANGELOG.md`.                             |
| **Conventional Commits**   | A commit message specification requiring the format `<type>(scope): <subject>`. Valid types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `security`, `privacy`. Used to automate release notes and CHANGELOG entries.                                             |
| **Error Budget Policy**    | The rule set governing team behaviour when the error budget is low or exhausted: feature work freezes, postmortems are mandatory, and only reliability improvements are deployed until the budget recovers.                                                               |
| **Mypy**                   | A static type checker for Python. Run in `strict` mode (`pyproject.toml`). All source files must pass mypy before merge. Type stubs for third-party libraries without inline types are listed in `[[tool.mypy.overrides]]`.                                               |
| **OpenAPI**                | The specification for describing REST APIs (formerly Swagger). FastAPI auto-generates an OpenAPI schema at `/docs`. Used alongside AsyncAPI — OpenAPI for REST, AsyncAPI for Kafka.                                                                                       |
| **postmortem**             | A blameless retrospective document written after every production incident or significant reliability event. Stored in `docs/postmortems/`. Required after every SLO breach and every manual override of the FeedbackLoop.                                                |
| **pre-commit**             | A framework for managing git pre-commit hooks. Configured in `.pre-commit-config.yaml`. Hooks run: Ruff (lint + format), Mypy (type check), Bandit (SAST), detect-secrets (credential scan). `--no-verify` is forbidden.                                                  |
| **PRR**                    | Production Readiness Review — a mandatory checklist completed and signed off by SRE Lead, Tech Lead, Security Lead, and DPO before every production deployment. Template at `docs/sre/prr/PRR-TEMPLATE.md`.                                                               |
| **Ruff**                   | A fast Python linter and formatter written in Rust. Replaces flake8, isort, and black. Configured in `pyproject.toml [tool.ruff]`. Enforces pycodestyle, pyflakes, isort, bugbear, comprehensions, pyupgrade, and security rules.                                         |
| **SDD**                    | Spec-Driven Development — no code is written without a referenced spec. Steps: write spec → ADR review → DPIA check → GitHub Issue → implement → test → update ADR → update CHANGELOG. Described in `skills/sdlc/spec-lifecycle.md`.                                      |
| **SDLC**                   | Software Development Lifecycle — the full process from requirements through design, implementation, testing, deployment, and maintenance. In this repo the SDLC is governed by the SDD cycle in `CLAUDE.md §2`.                                                           |
| **Agentic SDLC**           | The 13-phase lifecycle in which AI agents are first-class participants (Conception → … → Post-Deploy & Learn), governed by two-tier HITL: Spec-as-PR for pre-code artefacts, the runtime gateway for real-world effects. Reference: `docs/process/WORKFLOW.md`, ADR-0052. |
| **DoD**                    | Definition of Done — the global checklist a change must satisfy before it is complete (tests, ≥80% coverage, docs, guardrails, security). `docs/process/DEFINITION_OF_DONE.md`.                                                                                           |
| **DoR**                    | Definition of Ready — the criteria an issue must meet before entering a sprint (spec linked, NFRs, acceptance criteria, security review). `docs/process/DEFINITION_OF_READY.md`.                                                                                          |
| **DoR-Release**            | Definition of Release — the release-candidate readiness checklist before production promotion. `docs/process/DEFINITION_OF_RELEASE.md`.                                                                                                                                   |
| **Feature State Manifest** | `docs/product/FEAT-{id}/state.yaml` (schema `feature_state_v1`) — the compact, machine-readable record of a feature's current phase, approvals, gates passed, and allowed / prohibited agent actions (ADR-0054).                                                          |
| **FinOps**                 | Cloud financial operations — managing and attributing spend. Here: the LLM CI token-budget circuit breaker and cost alerting.                                                                                                                                             |
| **Maturity Model**         | The agentic maturity scale assessed by `make agentic-maturity-check`; gates advanced autonomy on demonstrated capability and governance (ADR-0040).                                                                                                                       |
| **Persona**                | A scoped behavioural profile in `.claude/personas/` (role, autonomy ceiling, skills, prohibited paths) that restricts — never expands — the default behavioural contract for non-engineering users.                                                                       |
| **Phase Gate**             | A machine-readable gate contract (`docs/process/gates/phase-gates.yaml`, schema `phase_gates_v1`) defining, per lifecycle phase, the required artifacts / approvals, CI checks, and allowed / prohibited agent actions (ADR-0054).                                        |
| **Quality Gate**           | A blocking CI check that must pass to progress (coverage ≥80%, zero CRITICAL SAST/SCA, PRR ≥90%, etc.); summarised per phase in `docs/process/WORKFLOW.md`.                                                                                                               |
| **RACI**                   | Responsible, Accountable, Consulted, Informed — the responsibility-assignment matrix for SDLC roles. `docs/process/RACI.md`.                                                                                                                                              |
| **Spec-as-PR**             | The governance model where pre-code artefacts (discovery, NFR, spec, ADR) are reviewed via GitHub PR — the human-review equivalent of the runtime HITL gateway for the spec phase (ADR-0052, `docs/process/HITL-GOVERNANCE.md`).                                          |

## 10. Python & Framework Stack

| Term                    | Definition                                                                                                                                                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **aiokafka**            | An async Python Kafka client built on top of kafka-python. Used in `src/workers/` for `RequestConsumer` and `RequestProducer`. Provides asyncio-native consumer groups and producer batching.                                         |
| **anthropic**           | The official Anthropic Python SDK. Used in `src/agents/llm_client.py` (`AnthropicLLMClient`) to call Claude models. Messages API; streaming and batch modes supported.                                                                |

| **fakeredis**           | A pure-Python in-memory Redis implementation used in unit tests to avoid requiring a running Redis instance. Drop-in replacement for the `redis` client in test fixtures.                                                             |
| **FastAPI**             | The async Python web framework used for the REST API layer. Auto-generates OpenAPI docs, enforces Pydantic request/response validation, and provides dependency injection via `Depends`.                                              |
| **httpx**               | An async-capable Python HTTP client. Used for service-to-service HTTP calls and for fetching Prometheus metrics in `FeedbackLoop`. Replaces `requests` throughout the codebase.                                                       |
| **Pydantic**            | A Python data validation library using type annotations. All API request/response schemas, config models (`pydantic-settings`), and event payloads are Pydantic models. Provides runtime validation and JSON serialisation.           |
| **pydantic-settings**   | A Pydantic extension for loading settings from environment variables and `.env` files. `Settings` class in `src/shared/config.py` is the single source of truth for all runtime configuration.                                        |
| **pytest-asyncio**      | A pytest plugin enabling `async def` test functions. `asyncio_mode = "auto"` is set in `pyproject.toml`, so all async tests run automatically without per-test `@pytest.mark.asyncio`.                                                |
| **slowapi**             | A rate-limiting library for FastAPI/Starlette backed by Redis. Protects public endpoints from abuse. Limits defined in `src/api/middleware.py`.                                                                                       |
| **SQLAlchemy[asyncio]** | SQLAlchemy with async driver support via `asyncpg`. Provides async sessions and ORM models. All DB interactions go through domain services using async context managers.                                                              |
| **structlog**           | A Python structured logging library. Binds contextual variables (trace_id, agent_id) once per request and propagates them to all nested log calls automatically. Emits JSON for machine parsing.                                      |
| **tenacity**            | A Python retry library. Used for resilient external calls (LLM API, Prometheus scrape, Kafka produce). Standard retry policy: exponential backoff, max 3 attempts, reraise on final failure.                                          |
| **Uvicorn**             | An ASGI server used to serve the FastAPI application. Configured with `uvicorn[standard]` (includes uvloop and httptools for performance). Used directly in development and wrapped by gunicorn in production.                        |
