# Database Migrations

> **Owner:** Tech Lead | **Tooling:** Alembic (`alembic/versions/`) | **Companion:** [`erd.md`](erd.md), [`data-model-catalog.md`](data-model-catalog.md)

All PostgreSQL schema changes go through Alembic. Migrations are **forward-only in production** and
reviewed via PR like any other change (SDD). The migration files in `alembic/versions/` are the
authoritative schema; `erd.md` and `data-model-catalog.md` summarise them.

## 2. Commands

```bash
uv run alembic upgrade head                       # apply all pending migrations
uv run alembic revision --autogenerate -m "msg"   # generate a migration from model changes
uv run alembic downgrade -1                        # roll back one revision (NON-prod / recovery only)
uv run alembic current                             # show the applied revision
uv run alembic history                             # show the full chain
```

`make setup` / `make setup-core` run `alembic upgrade head` after starting Postgres.

## 3. Authoring rules

- **One concern per migration**; give it a clear message. Review the autogenerated diff — Alembic
  does not detect every change (e.g. server defaults, CHECK constraints, index method).
- **Always implement `downgrade()`** so the migration is reversible in staging/recovery.
- **Encryption stays application-side.** L1/L2 PII columns are encrypted by `EncryptedField`
  (ADR-0018), not by the DB — a migration adds the `Text` column; the app encrypts the value. Never
  add a migration that would log or copy plaintext PII.
- **Indexes for access paths.** Every table here ships the indexes its queries need (see `erd.md`);
  add the matching index in the same migration as a new query path.

## 4. Backward-compatible (expand → migrate → contract)

Schema changes must not break the running version mid-deploy (canary runs old + new together):

1. **Expand** — add the new nullable column / table (deploy; old code ignores it).
2. **Migrate** — backfill + start writing the new column (deploy).
3. **Contract** — drop the old column once no code reads it (a later deploy).

A rename = add-new + backfill + switch-reads + drop-old across **three** migrations, never one. This
mirrors the API breaking-change rule (ADR-0024) at the data layer.

## 5. Retention

Redis data expires by TTL (see [`redis-key-naming.md`](redis-key-naming.md)). PostgreSQL retention
(ADR-0013) — e.g. operational logs ≤ 90d, audit logs per compliance — is an **operational** policy;
there is no automatic purge job in this template yet. When one is added (a `batch-jobs` CronJob is the
intended home), add its retention tests here and a row to `data-model-catalog.md`.

## 6. Backup & restore

Local snapshots: `infrastructure/scripts/db/backup.sh` / `restore.sh` (Postgres `pg_dump` + Redis
RDB). Production backup/DR cadence is an SRE responsibility — see the resilience plan (a future
`docs/resilience/` artefact, improvement-plan Wave 9). Backups of `audit_events`/`agent_memory_documents`
contain **encrypted** L1/L2 fields; the `DB_ENCRYPTION_KEY` must be backed up **separately** (Vault),
or restored data is unreadable.
